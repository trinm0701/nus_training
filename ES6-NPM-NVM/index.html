ES6 Overview

ES6 là bộ nền giúp JavaScript dễ viết ở quy mô project

ES6, hay ES2015, đưa vào nhiều tính năng mà các framework hiện đại dùng liên tục: let/const, arrow function, class, module, destructuring, default parameter, template literal, Promise và spread syntax.

Đọc code ES6 theo 4 câu hỏi

  • Dữ liệu đi đâu? destructuring, spread, rest parameter.
  • Function giữ ngữ cảnh nào? arrow function dùng lexical this.
  • Module xuất/nhập gì? named export, default export, side-effect import.
  • Giá trị thiếu được xử lý sao? optional chaining và nullish coalescing.
const user = {
  id: 7,
  profile: { name: "An", role: "admin" }
};

const label = `${user.profile.name} - ${user.profile.role}`;
const role = user.profile?.role ?? "guest";

export function canManage(account) {
  return account.permissions?.includes("manage") ?? false;
}

Điểm nên dùng mặc định

Ưu tiên const, chỉ dùng let khi cần gán lại. Viết function ngắn bằng arrow, tách module theo trách nhiệm, tránh nhồi mọi thứ vào một file.

Điểm không nên thần thánh hóa

ES6 không tự làm code sạch. Code vẫn cần tên biến rõ, boundary rõ, xử lý lỗi đầy đủ và dependency được khóa bằng lockfile.

Arrow Function, Classes, Spread

Ba cú pháp xuất hiện dày đặc trong codebase hiện đại

Arrow function

(items) => items.map(item => item.id)
  • Gọn cho callback: map, filter, reduce, event handler.
  • Không có this riêng; this lấy từ scope bên ngoài.
  • Không dùng làm constructor, không phù hợp khi method cần dynamic this.
const activeNames = users
  .filter((user) => user.active)
  .map((user) => user.name);

const team = {
  name: "Frontend",
  members: ["An", "Binh"],
  printLater() {
    setTimeout(() => {
      console.log(this.name);
    }, 100);
  }
};

Class

class là cú pháp rõ ràng hơn để tạo object có prototype chung. Dùng khi domain thật sự có entity, behavior và inheritance hợp lý. 

class HttpError extends Error {
  constructor(status, message) {
    super(message);
    this.status = status;
  }
}

Spread syntax

... mở array/object hoặc truyền danh sách argument. Rất hữu ích khi tạo bản copy mới thay vì sửa trực tiếp dữ liệu cũ. 

const nextUser = {
  ...user,
  profile: {
    ...user.profile,
    role: "admin"
  }
};

Class là mặt cú pháp, module là boundary của project

JavaScript vẫn là ngôn ngữ prototype-based. class, extends và super giúp mô tả inheritance dễ đọc hơn, nhưng object vẫn chia sẻ behavior thông qua prototype chain.

class

Khuôn behavior

Đặt constructor và method chung cho nhiều instance.

extends

Kế thừa có chọn lọc

Dùng khi subtype thật sự là một biến thể của base type.

module

Đóng gói API

Chỉ export phần cần dùng, giấu implementation detail.

Ví dụ OOP vừa đủ

Tránh tạo class chỉ để chứa function tĩnh. Với logic thuần, module function thường đơn giản hơn. Với entity có state và behavior đi cùng nhau, class giúp code dễ đọc. 

class Cart {
  #items = [];

  add(product, quantity = 1) {
    this.#items.push({ product, quantity });
  }

  get total() {
    return this.#items.reduce(
      (sum, item) => sum + item.product.price * item.quantity,
      0
    );
  }
}

Import/Export module trong ES6

ESM giống như mỗi file công khai một vài “cổng ra” bằng export, rồi file khác nối vào đúng cổng đó bằng import. Vì cú pháp import thường nằm cố định ở đầu file, browser hoặc bundler có thể đọc trước project cần những file nào trước khi chạy code.

Live binding nghĩa là import không copy giá trị một lần rồi thôi. Nó giống một đường link tới binding gốc: khi module gốc đổi giá trị export, nơi import đọc lại sẽ thấy giá trị mới.

Trước ES6

Code browser thường dùng biến global hoặc IIFE để tự tạo scope. Khi project lớn, thứ tự nhúng file bằng <script> rất dễ sai, dependency không hiện rõ và dễ đè tên biến trên window. 

<script src="./money.js"></script>
<script src="./cart.js"></script>

// cart.js phụ thuộc vào biến global Money
const total = Money.format(cart.total);

Với ES6 module

Dependency được khai báo ngay trong file bằng import. Bundler có thể phân tích graph module, chỉ đóng gói phần được dùng và browser hỗ trợ module bằng type="module". 

import { formatMoney } from "./money.js";

const total = formatMoney(cart.total);
// settings.js
export let theme = "light";

export function switchTheme(nextTheme) {
  theme = nextTheme;
}

// screen.js
import { theme, switchTheme } from "./settings.js";

console.log(theme); // "light"
switchTheme("dark");
console.log(theme); // "dark" vì theme là live binding

Named export

Dùng cho nhiều API rõ tên trong cùng module.

import { sum, avg } from "./stats.js";

Default export

Dùng khi module có một giá trị chính.

import createStore from "./store.js";

Side-effect import

Dùng khi module chỉ cần chạy setup.

import "./setup-analytics.js";

Dynamic import

Dùng khi chỉ muốn tải module lúc thật sự cần, ví dụ user mở trang Admin thì mới tải code của Admin. import() luôn trả về Promise. 

// Có await: chờ module tải xong rồi dùng ngay
async function openAdminPage() {
  const adminModule = await import("./admin-page.js");
  adminModule.renderAdminPage();
}

// Không await: nhận Promise, xử lý bằng .then()
function preloadAdminPage() {
  import("./admin-page.js").then((adminModule) => {
    adminModule.renderAdminPage();
  });
}

Practical Patterns

Pattern nên dùng khi đưa ES6 vào code thật

Module pattern hiện đại

Module tốt nên giống một “hộp công cụ nhỏ”: tên file nói rõ trách nhiệm, chỉ export những hàm bên ngoài thật sự cần, còn helper nội bộ giữ private trong file. Cách này giúp file khác không phụ thuộc vào chi tiết triển khai. 

// money.js - module chuyên xử lý tiền tệ
const currency = "VND";

function normalizeAmount(value) {
  return Number(value || 0);
}

export function formatMoney(value) {
  return normalizeAmount(value).toLocaleString("vi-VN") + "đ";
}

export function sumPrices(items) {
  return items.reduce((total, item) => total + normalizeAmount(item.price), 0);
}

// checkout.js
import { formatMoney, sumPrices } from "./money.js";

const total = sumPrices(cart.items);
console.log(formatMoney(total));
  • Nên export: API ổn định như formatMoney, sumPrices.
  • Không nên export: helper nội bộ như normalizeAmount nếu bên ngoài không cần biết.
  • Lợi ích: đổi logic bên trong module mà ít làm vỡ code import bên ngoài.

Async/await cải tiến gì?

Trước đây, code bất đồng bộ thường dùng callback hoặc chuỗi .then().catch(). async/await không làm Promise biến mất; nó chỉ giúp cách viết nhìn giống code tuần tự, dễ đọc hơn khi có nhiều bước phụ thuộc nhau. 

// Cách then/catch: ổn, nhưng dễ dài khi nhiều bước nối tiếp
getUser(userId)
  .then((user) => getOrders(user.id))
  .then((orders) => renderOrders(orders))
  .catch((error) => showError(error));

// Async/await: đọc từ trên xuống dưới
async function loadOrders(userId) {
  try {
    const user = await getUser(userId);
    const orders = await getOrders(user.id);
    renderOrders(orders);
  } catch (error) {
    showError(error);
  }
}

Async/await error handling

Với await, lỗi Promise reject sẽ đi vào catch giống lỗi đồng bộ. Nhưng vẫn cần xác định boundary: function nào chỉ throw tiếp, function nào log, function nào hiển thị message cho user. 

async function loadProfile(userId) {
  try {
    const profile = await getUser(userId);
    renderProfile(profile);
  } catch (error) {
    reportError(error);
    showToast("Không tải được hồ sơ.");
  } finally {
    hideLoading();
  }
}

Nên dùng async/await khi

  • Có nhiều bước cần chạy tuần tự.
  • Cần try/catch/finally rõ ràng.
  • Code nghiệp vụ cần đọc như một luồng xử lý.

Không nhất thiết dùng khi

  • Chỉ transform Promise một dòng đơn giản.
  • Cần chạy song song, nên dùng Promise.all.
  • Đang setup pipeline stream/event liên tục.

Fetch API wrapper

fetch chỉ reject khi lỗi network hoặc request bị chặn. Với HTTP 404/500, cần kiểm tra response.ok rồi tự throw lỗi phù hợp. 

export async function requestJson(url, options = {}) {
  const response = await fetch(url, {
    headers: { "Content-Type": "application/json" },
    ...options
  });

  if (!response.ok) {
    throw new Error(`HTTP ${response.status}`);
  }

  return response.json();
}

ESM vs CommonJS

Cả hai đều dùng để chia code thành module, nhưng khác ở thời điểm load, cú pháp và mức độ tối ưu cho tooling hiện đại.

Tiêu chí ESM CommonJS
Cú pháp import / export require / module.exports
Thời điểm phân tích Phân tích tĩnh trước khi chạy, import thường ở top-level. Load lúc runtime, require() có thể nằm trong if hoặc function.
Gom code khi build Dễ bỏ phần code không dùng khi build vì import/export rõ từ đầu. Khó bỏ code thừa hơn vì require() có thể được gọi linh hoạt khi chạy.
Môi trường Chuẩn hiện đại cho browser, Node.js mới và frontend bundler. Phổ biến trong Node.js cũ và nhiều package backend lâu đời.
Load động Dùng import("./file.js"), trả về Promise. require("./file.js") chạy đồng bộ trong CommonJS.
Khi nào chọn? Project mới, frontend, package muốn tối ưu bundle. Codebase Node.js cũ hoặc package/tooling chỉ hỗ trợ CommonJS.

Bundler là công cụ gom nhiều file thành bundle để chạy trên browser, ví dụ Vite, Webpack, Rollup. Tree shaking là bước loại bỏ phần export không được dùng để bundle nhẹ hơn.

Cách nhớ nhanh: code mới nên ưu tiên ESM; CommonJS vẫn cần hiểu vì còn xuất hiện nhiều trong Node.js package và project cũ.

NPM, YARN, PNPM / NVM toolbox

Toolbox giúp project chạy giống nhau trên mọi máy

Package manager quản dependency của project. NVM quản Node.js version. Hai phần này nên được cấu hình cùng nhau để giảm lỗi “máy em chạy, máy anh không chạy”.

NPM là gì?

npm là package manager đi kèm Node.js, dùng để cài package, chạy scripts và publish package lên npm registry. 

npm init -y
npm install axios
npm install -D vite
npm run dev

Package được cài ở đâu?

Cài local sẽ tạo node_modules/ trong project và ghi dependency vào package.json. Cài global đặt package ở global prefix của máy, thường dùng cho CLI dùng chung. 

npm install react
# ./node_modules/react

npm install -g serve
# global prefix/bin/serve

Hướng dẫn package.json

File package.json mô tả metadata, scripts, dependencies, devDependencies và package manager mà project mong muốn. 

{
  "name": "training-app",
  "type": "module",
  "scripts": {
    "dev": "vite",
    "test": "vitest"
  },
  "dependencies": {
    "react": "^19.0.0"
  },
  "devDependencies": {
    "vite": "^7.0.0"
  },
  "packageManager": "pnpm@10.0.0"
}

So sánh package manager

NPM, Yarn, PNPM khác nhau thế nào?

Cả ba đều cài package từ registry và chạy scripts trong package.json. Khác biệt chính nằm ở tốc độ, cách quản node_modules, lockfile và mức độ phù hợp với team/project. Tốc độ thực tế còn phụ thuộc version, cache, mạng, CI và độ lớn của dependency graph.

Tiêu chí npm Yarn pnpm
Điểm mạnh Có sẵn khi cài Node.js, dễ dùng, tài liệu và cộng đồng rất lớn. Từng nổi bật vì cài nhanh và lockfile ổn định; hiện vẫn phổ biến trong nhiều team frontend. Cài nhanh, tiết kiệm dung lượng nhờ store chung và symlink.
Vì sao Yarn có vẻ nhanh hơn npm? npm đời mới đã cải thiện nhiều, nhưng npm cũ thường chậm hơn ở bước resolve/cài dependency. Yarn classic từng nhanh hơn nhờ cache tốt, cài song song tốt hơn và lockfile nhất quán. pnpm thường nhanh vì package đã tải được tái dùng từ store chung thay vì copy lại nhiều lần.
node_modules Cài package trực tiếp vào node_modules của project. Tùy version/cấu hình; Yarn classic dùng node_modules, Yarn Berry có thể dùng Plug'n'Play. Dùng content-addressable store, rồi link package vào project.
Version/dòng sản phẩm npm đi theo Node.js, update độc lập được bằng npm. Yarn classic là Yarn v1; Yarn Berry là Yarn v2+ với nhiều thay đổi lớn như Plug'n'Play. pnpm là package manager riêng, thường dùng qua Corepack hoặc cài global.
Lockfile package-lock.json yarn.lock pnpm-lock.yaml
Ưu điểm khi dùng Ít setup nhất, phù hợp training, project nhỏ, hoặc team muốn dùng mặc định của Node.js. Phù hợp team đã chuẩn hóa Yarn hoặc monorepo cần workspace quen thuộc. Phù hợp monorepo, project nhiều package, CI cần cài nhanh và tiết kiệm cache.
Lưu ý Có thể chậm hơn pnpm ở project lớn; vẫn nên commit lockfile. Cần thống nhất version Yarn vì Yarn classic và Yarn Berry khác nhau khá nhiều. node_modules chặt chẽ hơn, có thể lộ lỗi import package chưa khai báo trong package.json.
Khi nào nên dùng? Khi muốn đơn giản, ít quy ước, onboarding nhanh. Khi project/team hiện tại đã dùng Yarn ổn định. Khi muốn tốc độ, tiết kiệm dung lượng, quản monorepo tốt.

Quy tắc thực tế: trong một project chỉ nên dùng một package manager và commit đúng một lockfile tương ứng.

Yarn classic là gì?

Yarn classic là Yarn v1. Đây là dòng Yarn cũ, rất phổ biến, mặc định vẫn tạo node_modules giống npm.

Yarn Berry là gì?

Yarn Berry là Yarn v2 trở lên. Nó thay đổi nhiều thứ so với Yarn v1, có thêm Plug'n'Play, plugin và cách cấu hình hiện đại hơn.

Plug'n'Play là gì?

Plug'n'Play, hay PnP, là cách Yarn Berry bỏ qua node_modules. Yarn lưu bản đồ dependency trong file cấu hình và chỉ đúng package nào được khai báo mới được import.

pnpm store và symlink là gì?

pnpm tải package vào một kho chung trên máy. Project chỉ link tới package trong kho đó, nên nhiều project dùng cùng version sẽ không phải copy package nhiều lần.

Checklist sau buổi học

Arrow function

Dùng cho callback và function nhỏ. Tránh dùng khi cần dynamic this hoặc constructor.

Class

Dùng cho entity có state + behavior. Với helper thuần, export function thường nhẹ hơn.

Fetch

Luôn kiểm tra response.ok, parse JSON ở một wrapper chung và xử lý lỗi ở boundary UI.
